# Plan: scrybe-mermaid-render + Drake-Swarm

## Context

We want to experiment with 100% agentic development of a new Rust crate
`scrybe-mermaid-render` — a pure-Rust reimplementation of mmdc (the Mermaid CLI
renderer) for the diagram types used most heavily in practice: **sequence
diagrams** and **flowcharts/directed graphs**.

The experiment has three interlocking deliverables:

1. **`scrybe-mermaid-render`** — the Rust crate with PyO3 bindings
2. **Trace test harness** — mmdc as the oracle, multi-tier grading (structural
   pass/fail + SSIM score)
3. **Drake-swarm** — an agentic orchestration system where Drake (agent identity,
   gnuc home server) runs phases and rounds, handing small goals to local
   inference models and routing results to Claude Code + Codex for grading

No existing Mermaid parsing or graph layout exists in the workspace. The
`scrybe-render` crate delegates Mermaid rendering to client-side Mermaid.js.
`scrybe-panels` and `scrybe-swarm` exist as stubs (~78 and ~67 LOC respectively)
in `experimental/` and form the foundation for the drake-swarm orchestration
layer.

### Design revision: SVG-first

Mermaid's native output is SVG — `mmdc` produces SVG and converts to PNG by
screenshotting via Puppeteer/Chromium. This crate follows the same model:

**SVG is the primary deliverable. PNG is a secondary conversion.**

Every SVG produced embeds its Mermaid source in a `<metadata>` element,
making it self-describing and round-trippable without a sidecar file.
This extends the convention established by `scrybe-mermaid` (PNG iTXt) to SVG.

```text
source → parse → layout → SVG  ←  primary deliverable
                            │       <metadata> embeds source + sha256
                            ↓  (optional `png` feature)
                        resvg/tiny-skia → PNG
```

SVG metadata format (namespace `https://scrybe.ai/ns/mermaid`):
```xml
<metadata>
  <scrybe:mermaid xmlns:scrybe="https://scrybe.ai/ns/mermaid">
    <scrybe:source><![CDATA[sequenceDiagram
  A->>B: Hello]]></scrybe:source>
    <scrybe:sha256>83af36...</scrybe:sha256>
  </scrybe:mermaid>
</metadata>
```

Consequences:
- Trace tests compare SVG structurally first (no mmdc oracle needed for Tier 1)
- SSIM grading rasterizes our SVG vs mmdc SVG — font differences tolerated at 0.85
- `render_to_svg(source)` is the primary Python API; `render_to_png` is opt-in

### Public API

```rust
/// Primary — returns SVG with <metadata> embedding source + sha256.
pub fn render_to_svg(source: &str) -> Result<String>;

/// Secondary — rasterizes render_to_svg output via resvg. Requires `png` feature.
#[cfg(feature = "png")]
pub fn render_to_png(source: &str) -> Result<Vec<u8>>;
```

PyO3 Python surface:
```python
from scrybe_mermaid_render import render_to_svg, render_to_png
svg_str   = render_to_svg("sequenceDiagram\n  A->>B: Hello")
png_bytes = render_to_png("graph TD\n  A --> B")  # requires png feature
```

### Sequence Diagram Scope (MVP)

Syntax to support:
- `participant`, `actor` declarations
- `->>`, `-->>`, `->`, `-->`, `-x`, `--x` message arrows
- `Note over`, `Note left of`, `Note right of`
- `loop`, `alt`, `opt`, `par` blocks (basic — no nested alt/opt)
- `activate` / `deactivate`

### Flowchart Scope (MVP)

Syntax to support:
- `graph TD | LR | BT | RL` direction
- Node shapes: `[]` (rect), `()` (rounded), `{}` (diamond), `([])` (stadium)
- Edge types: `-->`, `---`, `-.->`, `==>`, `--text-->`
- `subgraph` (render as grouped cluster, no nested layout)
- Node labels with quotes: `A["My Label"]`

Layout algorithm: **Sugiyama (layered graph drawing)**
1. Cycle removal (DFS-based)
2. Layer assignment (Longest Path)
3. Crossing minimization (Barycenter heuristic)
4. Coordinate assignment (Brandes-Köpf simplified)

Use `petgraph` for graph structure; implement layout steps manually (no existing
Rust dagre equivalent exists).

### Oracle Generation

`scripts/gen_oracle.sh` produces **SVG only** — PNG is not needed as an oracle
because both sides are rasterized by the same engine (resvg) for SSIM comparison.

```bash
bash scripts/gen_oracle.sh              # hand-written fixtures only
bash scripts/gen_oracle.sh --with-upstream  # + mermaid-cli test-positive/
```

Oracle SVG files are **gitignored** (reproducible from fixtures + mmdc).

### Grading: Three Tiers

**Tier 0 — Metadata round-trip** (always runs, gates test, no oracle needed):
- Verify `<scrybe:source>` is present in candidate SVG
- Verify the embedded source matches the original input verbatim
- Self-contained check — no oracle file required

**Tier 1 — Semantic element comparison** (gates test, requires oracle SVG):
- Parse mmdc oracle SVG: extract element counts + visible text labels
  - Sequence: `rect.actor` count, `line`/arrow count, `text.messageText` labels
  - Flowchart: `g.node` count, `g.edgePaths` count, `text.label` labels
- Compare against candidate SVG extraction
- Counts and label sets must match (within tolerance)

**Tier 2 — SSIM visual score** (not gating, requires oracle SVG + `png` feature):
- Rasterize oracle SVG with resvg → pixels
- Rasterize candidate SVG with resvg → pixels
- Compute SSIM — both sides use the same renderer, so threshold is **0.92+**
  (was 0.85 when comparing Chromium vs resvg; now apples-to-apples)
- Score recorded in CalibrationLog; Drake uses it to decide round pass/fail

### Grader API (`tests/common/grader.rs`)

```rust
pub struct GradeResult {
    pub fixture: String,
    pub metadata_pass: bool,           // Tier 0
    pub structural_pass: Option<bool>, // Tier 1 (None if no oracle)
    pub ssim: Option<f64>,             // Tier 2 (None if no oracle or no png feature)
    pub notes: Vec<String>,
}

pub fn grade_sequence(fixture: &str, source: &str,
                      candidate_svg: &str, oracle_dir: &Path) -> GradeResult;
pub fn grade_flowchart(fixture: &str, source: &str,
                       candidate_svg: &str, oracle_dir: &Path) -> GradeResult;
```

## Deliverable 3: Drake-Swarm Orchestration

### Overview

Drake (agent identity: `Drake (gnuc home server, <model>)`) orchestrates the
implementation project. The swarm uses:

- **Worker agents** — local Ollama models (qwen3-coder:30b or similar) receive
  small, scoped implementation goals
- **Graders** — Claude Code reviews code quality/style; Codex reviews
  algorithmic correctness vs oracle
- **Drake** — reads phase config, dispatches goals, collects grades, advances

### Phase Config Schema

New file: `scrybe-mermaid-render/drake-swarm.toml`


### Drake-Swarm Loop (per round)

```
1. Drake reads next goal from current phase
2. Drake sends goal to worker (Ollama) via McpClient
3. Worker produces code + tests
4. Drake sends code to Claude Code grader: "Review for Rust style, correctness, test coverage"
5. Drake sends code + trace test results to Codex grader: "Does this implement the goal correctly?"
6. Drake records RoundResult in CalibrationLog
7. If structural_pass AND ssim >= threshold: advance to next goal
8. Else: refine goal with grader feedback, retry (max N rounds)
9. When all goals in phase pass: advance to next phase
```

## Open Questions / Risks

1. **Sugiyama layout complexity** — Crossing minimization is NP-hard; the
   Barycenter heuristic is a well-known approximation but still ~300 LOC of
   careful Rust. This is the highest-risk phase. Drake should budget 3-5 rounds.

2. **mmdc anti-aliasing** — mmdc uses Chromium's text rendering; our resvg
   output will have different anti-aliasing. SSIM threshold of 0.85 accounts
   for this; pixel-exact match is not the goal.

3. **Oracle generation in CI** — mmdc requires Node.js + npm. CI workflow will
   need a setup step or pre-committed oracle files for a subset of fixtures.

4. **Drake on gnuc** — Drake runs on gnuc (home server). Ollama model must be
   installed there. The `scrybe-mcp-client` Ollama preset points to `:11434`
   by default; gnuc's IP/hostname must be configured in `agents.toml`.
